## 三、鉴权签名（sign）生成规则

签名是保障接口调用安全的核心参数，采用MD5加密算法生成，必须为**32位小写英文字符**，生成步骤及注意事项如下：

### 1. 标准生成步骤

**Step1**：字符串拼接

按照固定顺序拼接参数：`appid + 待翻译文本(q) + 随机数(salt) + 平台分配密钥`，拼接过程中不添加任何额外字符、空格，得到原始拼接字符串。

**Step2**：MD5加密

对Step1得到的拼接字符串做MD5加密，直接获取32位小写加密结果，即为最终签名sign。

### 2. 关键避坑注意事项

- 生成签名拼接字符串时，**待翻译文本q绝对不能做URL encode**，这是导致54001签名错误的最主要原因

- 签名生成完成后，发送HTTP/HTTPS请求时，必须对q参数做URL encode处理

- MD5加密结果必须转为小写，大写字符会直接导致鉴权失败

- 密钥需严格保密，不可泄露至第三方平台，避免账户被盗用

|错误码|错误含义|解决方案|

|52002|系统内部错误|接口临时波动，等待几秒后重新发起请求|
|52003|未授权用户|核对APPID是否正确，确认翻译服务已成功开通|
|54000|必填参数为空|逐一核查必填参数，排查漏传、误传、空值问题|
|54001|签名错误|检查签名拼接顺序、MD5是否32位小写、q是否提前URL编码|

## 七、完整接入示例

```Plain Text
from = 'en'
```

### 2. 签名生成过程

```Plain Text
# 拼接字符串
sign_str = appid + q + salt + appkey
# 结果：2015063000000001apple654781234567890

# MD5加密生成32位小写签名
sign = md5(sign_str)
# 结果：a1a7461d92e5194c5cae3182b5b24de1
```

### 3. 完整GET请求URL

（请求前已对q做URL encode）


## 八、接入核心注意事项总结

1. 全程严格使用UTF-8编码，避免乱码与鉴权失败

2. 签名生成是核心，切记q参数“签前不编码，请求才编码”

3. 目标语言to参数禁止设置为auto，否则请求失败

4. 小语种翻译需升级企业尊享版，个人账号勿尝试调用

5. 妥善保管APPID和密钥，禁止第三方平台泄露，防止IP封禁与账户盗用
