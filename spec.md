# AI 文件管理器 - 需求规格文档

## 项目概述

开发一个 Windows 桌面应用程序，用于集中管理和整理用户电脑上的 AI 相关配置文件（rules、agents、skills、commands、plugins 等）。

**技术栈**: Electron + React

## 核心功能

### 1. 类别管理系统

#### 1.1 类别定义
- 类别是用户可自定义的分类方式（如 rules、agents、skills、commands 等）
- 支持增删改类别操作
- 每个类别包含以下配置：
  - **类别名称**: 显示在界面上的名称
  - **展示类型**: 文件夹形式 或 文件形式
  - **搜索路径**: 支持配置多个搜索路径
  - **描述文件**: 如果是文件夹形式，需指定从哪个文件读取 description（如 `skill.md`）

#### 1.2 展示类型说明

**文件夹形式**:
- 一个文件夹代表一个资源
- 文件夹名称作为资源名称
- 从指定的配置文件中读取 description（如 `skills/skill.md`）
- 如果配置文件不存在，使用文件夹名称作为 description

**文件形式**:
- 一个 `.md` 文件代表一个资源
- 递归扫描配置路径下的所有 `.md` 文件
- 文件名（不含扩展名）作为资源名称
- 从文件的 YAML front matter 中读取 description

### 2. 文件扫描规则

#### 2.1 YAML Front Matter 解析
所有 `.md` 文件使用 YAML front matter 格式定义元数据：

```markdown
---
description: 这是资源的描述信息
---

# 文件内容
```

#### 2.2 扫描逻辑

**文件夹形式扫描**:
- 扫描配置路径下的一级子文件夹
- 读取每个文件夹下指定的配置文件（如 `skill.md`）
- 解析 YAML front matter 获取 description
- 容错处理：
  - 如果配置文件不存在，使用文件夹名称作为 description
  - 如果配置文件存在但无 YAML front matter，使用文件夹名称作为 description
  - 如果 YAML 解析失败，记录错误日志并使用文件夹名称作为 description

**文件形式扫描**:
- 递归扫描配置路径下的所有 `.md` 文件
- 包括多层级目录结构（如 `/rules/common/*.md`、`/rules/Python/*.md`）
- 解析每个文件的 YAML front matter 获取 description
- 容错处理：
  - 如果 `.md` 文件无 YAML front matter，使用文件名（不含扩展名）作为 description
  - 如果 YAML 解析失败，记录错误日志并使用文件名作为 description

#### 2.3 描述翻译
- 如果 description 为非中文，自动翻译成中文展示
- 保留原始 description 作为备用信息

### 4. 搜索功能

#### 4.1 全局搜索
- 搜索范围：所有类别的所有资源
- 搜索字段：
  - 资源名称（文件夹名/文件名）
  - 描述信息（中文和原文）
  - 文件路径
- 搜索方式：模糊匹配
- 实时搜索：输入时即时过滤结果

#### 4.2 搜索结果展示
- 高亮匹配的关键词
- 显示资源所属类别
- 保持与资源列表相同的卡片样式

### 5. 数据持久化

#### 5.1 配置文件结构

应用配置存储在 JSON 文件中：

```json
{
  "categories": [
    {
      "id": "skills",
      "name": "Skills",
      "displayType": "folder",
      "descriptionFile": "skill.md",
      "searchPaths": [
        "C:\\Users\\...\\.claude\\skills"
      ]
    },
    {
      "id": "rules",
      "name": "Rules",
      "displayType": "file",
      "searchPaths": [
        "C:\\Users\\...\\.claude\\rules"
      ]
    }
  ]
}
```

**默认配置**:

首次启动时，应用会在程序同级目录自动创建 `config.json`，包含以下默认配置：

```json
{
  "categories": [
    {
      "id": "default",
      "name": "默认类别",
      "displayType": "file",
      "searchPaths": []
    }
  ]
}
```

**注意**: 默认配置的搜索路径为空，用户需要通过设置界面添加搜索路径后才能扫描和显示资源。用户可以添加和配置自己的类别。

#### 5.2 缓存机制
- 扫描结果缓存到内存
- 点击刷新按钮时重新扫描
- 应用启动时自动扫描一次

### 6. 技术实现要点

#### 6.1 技术栈
- **前端框架**: React
- **桌面框架**: Electron
- **UI 组件库**: Ant Design 或 Material-UI（轻量美观）
- **状态管理**: React Context 或 Zustand
- **文件操作**: Node.js fs 模块
- **YAML 解析**: js-yaml
- **翻译服务**: 集成翻译 API（如 Google Translate API 或本地翻译库）

#### 6.2 核心模块

**文件扫描模块** (`scanner.js`):
- 递归扫描目录
- 解析 YAML front matter
- 返回资源列表

**翻译模块** (`translator.js`):
- 检测语言
- 翻译非中文描述
- 缓存翻译结果

**配置管理模块** (`config.js`):
- 读写配置文件
- 验证配置有效性
- 首次启动时自动创建默认配置文件
- 配置文件路径：便携版使用 `process.cwd()` 获取程序同级目录


**文件操作模块** (`fileOps.js`):
- 在文件管理器中定位文件
- 使用 `shell.showItemInFolder()`

### 7. UI/UX 设计原则

- **简洁**: 界面清爽，无冗余元素
- **美观**: 现代化设计风格，合理的间距和配色
- **轻量**: 快速启动，流畅交互
- **直观**: 功能一目了然，操作符合直觉
- **中文化**: 所有界面文字使用中文

### 8. 非功能性需求

#### 8.1 性能要求
- 应用启动时间 < 3 秒
- 扫描 1000 个文件 < 5 秒
- 搜索响应时间 < 500ms

#### 8.2 兼容性
- Windows 10 及以上版本
- 支持高 DPI 显示

#### 8.3 错误处理
- 路径不存在时提示用户
- 文件解析失败时跳过并记录日志
- 翻译失败时显示原文

### 9. 实现优先级

**P0 (核心功能)**:
1. 类别管理（增删改查）
2. 文件扫描（文件夹形式 + 文件形式）
3. YAML front matter 解析
4. 资源列表展示
5. 点击定位到文件管理器
6. 配置持久化
7. 刷新功能
8. 描述翻译（自动翻译非中文描述）

**P1 (重要功能)**:
1. 全局搜索
2. 翻译结果缓存
3. 高级搜索过滤

**P2 (优化功能)**:
1. 缓存机制优化
2. 错误处理优化
3. UI 美化
4. 性能优化

## 技术风险与应对

### 风险1: 翻译 API 调用频率限制
**应对**:
- 使用本地翻译库（如 translate.js）
- 缓存翻译结果
- 提供手动翻译开关

### 风险2: 大量文件扫描性能问题
**应对**:
- P0 阶段：使用同步扫描，满足 1000 个文件 < 5 秒的性能要求
- P2 优化：使用 Worker 线程进行扫描，提升大规模文件处理性能
- 分批加载和渲染
- 添加加载进度提示

### 风险3: 不同格式的 YAML front matter
**应对**:
- 使用健壮的 YAML 解析库
- 添加格式验证和容错处理
- 提供配置文件模板
